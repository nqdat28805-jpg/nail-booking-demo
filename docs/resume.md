## Step 3 Booking Data Bug Fixed
- Root cause found: `fetchSharedBookingContext` correctly accepted `activeHoldSlot`, but `buildPublicBookingContext` internally failed to pass this starting time down to `durationService.estimateDuration`. As a result, the `estimatedEndTime` field in `sharedContext.durationEstimate` was always `null`. When `BookingExperience` saved this draft to `sessionStorage`, `endTime` incorrectly became `null`, triggering the UI fallback in Step 3 which expects `endTime` to be present.
- Files changed:
  - `src/application/services/duration-service.ts`
  - `src/domain/availability/contracts.ts`
  - `src/application/services/availability-service.ts`
  - `src/server/public-booking.ts`
- What data path/storage path was fixed: Fixed `estimateDuration` to accept an optional `startTime` parameter so that `estimatedEndTime` is correctly evaluated and passed to the frontend to restore the correct persistent draft state.
- How the bug was verified: Verified that passing `startTime` to the service calculation correctly evaluates `addMinutesToTime`. Verified with `npm run build` that the project successfully compiles.
